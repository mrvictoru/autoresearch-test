Auto-research: Building a Ratcheting Experiment Harness in Python.

How we move beyond chaotic manual chat sessions to programmatically orchestrate LLM agents inside repeatable, sandboxed experiment loops.

Orchestration Framework

1.Define Programmatic Interfaces: Split the codebase into fixed environments and a single mutable candidate contract.
2.Orchestrate with Python: Candidate loading via dynamic imports, automated verification runners, and isolation barriers.
3.Score on Stable Simulators: Run deterministic simulation scenarios with pinned seeds to eliminate random metric variance.
4.Auto-Ratchet Mutations: Commit, evaluate, and rollback automatically—ratcheting forward only when scores strictly improve.
This shifts the engineer's role from writing and debugging local snippets to designing high-fidelity sandboxes and rigorous scoring pipelines.

Why it matters for Python Devs

  • Why Simulation Wins: Real-world testing telemetry contains too much noise; a simulation sandbox allows instant, high-fidelity, and stable scalar scoring.
  • Dynamic Exploration: Let AI autonomously explore thousands of parameter variations, models, or configurations while you maintain absolute discipline.
  • Auditable Ledger: Every accepted candidate is logged as a git commit and appended to a transparent TSV/JSON optimization ledger.

Framing: The path from vibe coding to reproducible auto-research.

Rather than treating AI as a human replacement that guesses answers, we treat it as an optimizer that runs validated trials under custom Python contracts.

1. Vibe Coding

Manual prompts, copy-pasting code, and informal visual verification.

Tradeoff: Highly subjective, zero telemetry, prone to silent regression, and low reproducibility.

2. Agentic Engineering

Agents running shell tools directly, writing arbitrary scripts or touching files across many directories.

Tradeoff: Fragile command sequences, high risk of workspace state pollution, and hard to replicate safely.

3. Auto-Research Harness

Isolated sandbox runtime where the LLM is restricted to modifying exactly *one* explicit Python module interface.

Tradeoff: Completely reproducible, sandboxed, and easily logged via standard execution streams.

The developer designs the boundary; the harness manages disciplined, automated search.

Ensuring Reproducibility

Harness design: Preventing AI cheating & statistical noise.

1. Deterministic Scenarios

We pin custom random states utilizing random.Random(seed) combined with isolated lists of training/validation seed constants in tasks.py, ensuring results are fully reproducible.

2. Subprocess Hygiene

Running successive candidates in isolated Docker subprocesses gets rid of system state pollution, module caches (sys.modules), and stale memory leakages.

3. Git-Diff Security Guard

To prevent the AI from cheating by editing evaluation constraints or tests, we check changes against git diff --name-only, rejecting any commit that alters non-policy files.

4. UnitTest Pre-checks

Run standard library unittest suites to guarantee candidate syntax and interface returns are fully sane before starting high-overhead simulations.

Immutable vs. Mutable Code Boundaries

  • Immutable Task Harness (tasks.py): Controls simulation rules, resource constraints, perishability equations, and seed lists.
  • Mutable Candidate Script (restaurant_train.py): The single approved surface of modification allowed for our LLM optimizer.

Eliminating Random Luck

  • If scenarios change dynamically or have random seeds, the AI could "win" simply from lucky seed draws.
  • Fixed scenario environments guarantee that every single score shift directly reflects actual logical improvements.
A hard statistical ratchet requires complete, unpolluted sandbox determinism.

Maintains a flat file tracking all candidate executions, SHA records, resulting scores, andkeep / discard outcomes.

The Duck-Typed Policy Contract

  • The Hook: Candidate files must expose exactly one function: build_policy().
  • The Interface: Returns an object implementing decide_orders(observation) and optional fit(scenarios, task).
  • Decoupled Logic: The evaluator runs this logic cleanly without knowing whether it is a heuristic, rule-base, or Neural Net.

The Target Challenge: The Restaurant Inventory Policy

A complex, multi-variable daily logistics environment that standardizes performance gains.

Simulator Core Design & Constraints

  • Dynamic Daily Demand: Overnight sequences of menu demands with varying customer volumes.
  • Multi-Ingredient Overlaps: Individual menu items require variable amounts of shared ingredients.
  • Physical Realities: Overlapping lead times, strict physical storage caps, and raw ingredient shelf-life perishability.
  • The Objective: Maximize restaurant service rate (sales) while minimizing ingredient expirations and holding costs.

Evolution of Policy Logic

1. Baseline Adaptive Heuristic
Blends weekday historical averages with recent usages. Uses square-root safety buffers & FIFO expiration tracking.
2. Proportional Shared Capacity Budgeting
Heuristics order without safety limits; our budget allocator scales ordered volumes down dynamically to fit the physical warehouse headroom.
3. Focused Residual Neural Hybrid (Best)
An MLP regressor predicts target margins (derived from a perfect-knowledge hindsight oracle), adding learned positive adjustments to cheese, onion, lettuce, and pasta.
The agent's sole task is rewriting restaurant_train.py to maximize operating income.

Practical Python lessons from building ratchetting sandboxes.

Key Python Engineering Lessons

  • Avert State Contamination: Dynamic module updates can pollute sys.modules caching or global singletons. Running iterations inside fresh subprocesses or isolated container pipelines ensures total runtime environment hygiene.
  • Resilient Environment Design: Gracefully handle host landscape mismatches (e.g. NVIDIA runtime discrepancies in local Docker contexts) with seamless CPU fallbacks to ensure un-halted pipelines.
  • The Power of the Ratchet: Bounded, statistical optimization feedback loops beat chat posturing and chaotic prompt-guessing every single time.

Architectural Safeguards

  • The task engine simulator remains 100% immutable.
  • The dynamic loading interface contract is heavily explicit.
  • The frontier progress is fully peer-reviewed and auditable.

Core Presentation Takeaway

Reliable AI integration is not "let the model code harder." It is engineering a meticulous loop in Python, setting strict boundaries, and letting structured validation compound improvements.

Engineering-first AI orchestration provides discipline, safety, and reproducible value.
