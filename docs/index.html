Autoresearch-Test

An open-source framework that uses a local LLM as a research agent to iteratively improve task-specific models.

60-minute workshop — press → or click Next to begin

What is Autoresearch?

  • Inspired by Andrej Karpathy's autoresearch project
  • Idea: let an AI (LLM) act as a research assistant that suggests what experiment to run next
  • Original version trains language models — we generalise it to any task
  • Examples in this repo: restaurant inventory management & blackjack

The Core Feedback Loop

┌──────────────┐ suggestion ┌───────────┐ new state ┌──────────────┐ │ Research │ ──────────► │ Trainer │ ──────────► │ Task │ │ Agent (LLM) │ │ │ │ Evaluator │ └──────┬───────┘ └───────────┘ └──────┬───────┘ ▲ │ │ score + context │ └──────────────────────────────────────────────────────┘

Each iteration: Agent proposes → Trainer applies → Evaluator scores → repeat.

Key Concept 1 — Research Agent

  • Receives the current model state and task context
  • Returns a plain-text suggestion (e.g. reorder_point=24 target_stock=55)
  • Backed by a local LLM (any OpenAI-compatible server)
  • Can also be a simple stub for testing — no GPU required

Key Concept 2 — Tasks

  • A Task defines the problem we want to solve
  • It provides:
    • name — unique identifier
    • describe_context() — what the LLM needs to know
    • initial_model_state() — starting parameters
    • evaluate_model(state) — returns a score (higher = better)
  • The framework never looks inside the task — black box

Key Concept 3 — Trainers

  • Sit between the LLM suggestion and the next model state
  • train_step(model_state, suggestion, task_context) → new_state
  • Responsible for parsing the LLM text and validating values
  • One trainer per task; stored in a TrainerRegistry

Key Concept 4 — The Runner

  • AutoresearchRunner glues everything together
  • Call runner.run(task, iterations=N)
  • Tracks the best score and full history of every iteration
  • Returns a RunResult dataclass you can inspect or log

Example Task — Restaurant Inventory

Scenario

A restaurant orders ingredients daily. Too much → waste. Too little → stockouts and lost revenue.

Model Parameters

  • reorder_point — stock level that triggers a new order
  • target_stock — how much to order up to

Score

Negative total cost over 14 simulated days (higher = less cost = better).

Example Task — Blackjack

Scenario

A simplified blackjack game with weighted card draws.

Model Parameter

  • hit_threshold — player hits when hand total is below this value (clamped 12–20)

Score

Mean reward over 300 rounds (+1 win, 0 draw, −1 loss).

Why a Local LLM?

  • Privacy — data never leaves your machine
  • Cost — no API fees; run as many iterations as you want
  • Reproducibility — pin the exact model version
  • Flexibility — swap models (Mistral, Llama, Phi, …) without code changes
  • Compatible with llama.cpp, Ollama, vLLM, LM Studio, etc.

        trainer = self.registry.get(task.name)
        model_state = task.initial_model_state()
        best_score = task.evaluate_model(model_state)

Putting It All Together

from autoresearch import *
from autoresearch.agent import LocalLLMResearchAgent

agent = LocalLLMResearchAgent(
    endpoint="http://localhost:8080",
    model="mistral-7b",
)

registry = TrainerRegistry()
registry.register("restaurant_inventory", InventoryPolicyTrainer())
registry.register("blackjack", BlackjackPolicyTrainer())

runner = AutoresearchRunner(agent=agent, registry=registry)
result = runner.run(RestaurantInventoryTask(), iterations=10)
print(result.best_model_state, result.best_score)

Extending the Framework

Three things you can customise:

  1. Prompt — change the system/user prompt in the agent
  2. Task — add any new domain by implementing ResearchTask + TaskTrainer
  3. Model architecture — swap dict-based state for neural-network checkpoints (Transformer, Mamba, etc.)

Swapping to a Neural-Net Model

  • Replace dict state with a checkpoint path
  • Trainer loads the model, applies hyperparameters from the LLM, and re-trains
  • Evaluator loads the checkpoint and computes a metric
  • Works the same for Transformer (attention) or Mamba (SSM) architectures
  • The runner doesn't change at all — it only passes dicts and floats

Recap

  1. Agent proposes new parameters using a local LLM
  2. Trainer parses and validates those parameters
  3. Task evaluator scores the resulting model
  4. Runner repeats and tracks the best result

Everything is pluggable — swap any component without touching the others.
