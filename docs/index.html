Auto-research
Overview
Overview

Auto-research turns model iteration into a repeatable research loop.

Instead of prompting for isolated edits, the agent runs bounded experiments, keeps strict improvements, and leaves an auditable frontier.

Core idea

1.Human defines the objective and the allowed surface.
2.Agent proposes and implements a small experiment.
3.Evaluator scores the result under a fixed benchmark.
4.Only better outcomes ratchet forward.
Auto-research matters because it converts trial-and-error into a closed loop with explicit constraints.

Why it matters

  • More experiments can run than a manual loop can support.
  • Human effort shifts from execution to framing and review.
  • Every accepted improvement is visible in the frontier history.

Framing
From prompt to process

The progression is vibe coding → agentic engineering → auto-research.

The human role moves upward: from writing edits, to supervising tools, to setting the research frame and judging the frontier.

Vibe coding

Ask for code, review it, and steer each edit manually.

Agentic engineering

Use tools and automation, but keep the human in the loop at each step.

Auto-research

Set the rules once, then let the agent iterate inside the sandbox.

The human sets direction; the system handles disciplined iteration.
Method
Ratchet loop
The loop

Propose. Run. Score. Keep. Repeat.

1. Propose

Read the brief and form a concrete hypothesis.

2. Run

Apply the change and execute the same evaluator.

3. Score

Use one scalar objective to decide whether the change wins.

4. Ratchet

Keep strict improvements; revert everything else.

The three-part structure

  • program.md: goal and constraints
  • immutable evaluator and benchmark
  • one mutable training or policy file

Why it is reliable

  • fixed benchmark
  • narrow edit surface
  • frontier log of every attempt
Auto-research is powerful because autonomy is paired with a ratchet.
This repository
Implementation
Repo implementation

This repository instantiates the loop on a restaurant inventory benchmark.

The benchmark is fixed. The agent gets one approved place to change behavior.

Core files

autoresearch/tasks.py
Immutable simulator: demand, perishability, lead times, storage limits, and shared ingredients.
autoresearch/experiments/restaurant_eval.py
Immutable evaluator: loads the policy and reports the score and metrics.
autoresearch/experiments/restaurant_train.py
Mutable surface: build_policy() returns the policy being tested.

Operational contract

  • Read program.md first.
  • Edit only restaurant_train.py.
  • Append outcomes to results.tsv.
docker compose run --rm autoresearch python -m autoresearch.experiments.restaurant_eval \ --experiment autoresearch/experiments/restaurant_train.py ./scripts/run_once.sh "short experiment note"

Same evaluator, same benchmark, comparable outcomes.

One simulator. One evaluator. One mutable policy.

Takeaways
Lessons
What the repo teaches

The winning pattern is constraint-first autonomy.

What actually moved the score

  • Better demand coverage for long lead-time ingredients
  • Budget-aware ordering under shared warehouse capacity
  • Selective hybridization after the baseline heuristic was already strong

Why this is a strong demo

  • The benchmark is fixed.
  • The mutation surface is explicit.
  • The frontier is visible and auditable.

Plain-language takeaway

Auto-research is not “let the model guess harder.” It is define the right loop, then let the system compound improvements.

Good automation comes from structure, not from removing discipline.
