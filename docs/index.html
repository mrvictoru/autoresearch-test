From autoresearch to multi-agent autoresearch

A plain-language walkthrough for people who want to use an agent harness without needing to track every recent LLM trend.

The simple idea

Autoresearch treats model-assisted coding like a repeatable experiment loop:

  • Read the rules of the benchmark.
  • Change one safe thing in the allowed file.
  • Run the evaluator the same way every time.
  • Keep only better results and log everything else.
You can think of it as scientific method for code changes: hypothesis → experiment → score → keep or discard.

Why people care

  • It reduces random trial-and-error.
  • It makes agent work auditable and reproducible.
  • It gives teams a way to let agents iterate without giving them unlimited freedom.

Evolution

Why it evolved into a multi-agent system

Single-agent autoresearch is strong for one experiment at a time. Multi-agent autoresearch adds a control plane so several specialized assistants can coordinate safely.

Single-agent mode

  • One assistant proposes, edits, runs, and judges.
  • Works well for focused iteration.
  • Starts to bottleneck when you want parallel experiments or better review.

Multi-agent mode

  • Planner suggests hypotheses.
  • Reviewer rejects duplicates or weak ideas.
  • Experiment workers run isolated trials.
  • Reporter + memory keeper summarize progress and record lessons.

What changed in practice

  • The benchmark stayed the same.
  • The orchestration around it became smarter.
  • Worktrees let many workers edit the same mutable file without collisions.

Key mental model

“Agent” here does not mean magic intelligence. It means a tool-driven helper with a narrow job, explicit rules, and a clear handoff.

Why this matters to non-experts

You do not need to understand every model family. The important part is the operating system around the agents: permissions, isolation, logging, and evaluation discipline.

This repository

How the code in this repo works

This repository is intentionally narrow: it is a harness-only restaurant inventory benchmark with a fixed simulator, a fixed evaluator, and one approved place for experiment changes.

Benchmark core

autoresearch/tasks.py
Immutable restaurant simulator: demand, perishability, lead times, storage limits, shared ingredients.
autoresearch/experiments/restaurant_eval.py
Immutable evaluator: loads the policy, optionally fits it, runs the task, prints metrics.
autoresearch/experiments/restaurant_train.py
Mutable policy file: the experiment surface. It must expose build_policy().

Control plane and bookkeeping

  • autoresearch/frontier.py: manages branch, commit, revert, and results.tsv.
  • autoresearch/control_plane.py: planner/reviewer/worker/reporter helpers.
  • scripts/run_once.sh: one atomic experiment attempt.
  • scripts/control_plane.sh: local multi-agent CLI.
  • scripts/run_worker.sh: runs one isolated worker attempt.
The design goal is control-plane upgrade, not benchmark rewrite.

What stays fixed

  • Docker-first execution path
  • Evaluator contract and metric format
  • Restaurant simulation rules
  • results.tsv as the frontier ledger

What changes during research

  • The policy logic in restaurant_train.py
  • Hypotheses in the local control plane
  • Worker notes, logs, and kept/discarded outcomes
In short: the repo gives agents a safe playground with strong boundaries.
Usage

How to use this repo with an agent harness

The workflow is intentionally repetitive. That is a feature, not a bug: it keeps experiments comparable and easy to audit.

1. Read the rules

Start with program.md, README.md, and docs/guide.md so the agent knows what is mutable and what is not.

2. Run the baseline

Run the evaluator before changing anything so the agent has a reference score and known-good behavior.

3. Try one change

Edit only autoresearch/experiments/restaurant_train.py for benchmark experiments.

4. Keep or discard

Run the evaluator, parse the score, append to results.tsv, and keep only strict improvements.

Useful commands

docker compose run --rm autoresearch python -m unittest discover -s tests -v docker compose run --rm autoresearch python -m autoresearch.experiments.restaurant_eval \ --experiment autoresearch/experiments/restaurant_train.py ./scripts/run_once.sh "short experiment note"

Multi-agent control plane commands

./scripts/control_plane.sh init --max-workers 2 ./scripts/control_plane.sh add-idea --title "Lead-time buffer" \ --hypothesis "Increase safety stock for long lead ingredients" \ --policy-family "heuristic reorder tuning" \ --rationale "Reduce stockout penalties" ./scripts/control_plane.sh review ./scripts/control_plane.sh launch-worker --idea-id <idea-id> ./scripts/run_worker.sh --worker-id <worker-id> --message "test one hypothesis" ./scripts/control_plane.sh status
Small controlled loops beat big untracked changes.
Takeaways

What to remember

If you are new to agent harnesses, the most important lesson is that success usually comes from good process design, not from chasing the newest model buzzword.

Autoresearch in one sentence

Use agents to run disciplined, logged, score-driven software experiments.

Multi-agent autoresearch in one sentence

Split that loop into specialized roles so multiple experiments can be proposed, reviewed, executed, and summarized safely.

Why this repo is a good teaching example

  • It has a clear boundary between mutable and immutable code.
  • It uses Docker for repeatability.
  • It keeps a plain-text frontier ledger.
  • It shows how a control plane can grow around a stable benchmark.

Best audience takeaway

You do not need “AI magic.” You need a workflow that makes automated changes safe to try, easy to compare, and cheap to reject.
